Scenario: Northern Trail Outfitters needs to set up a Data Extension to store customer information that will be used as the sending audience for email campaigns. The data includes customer ID, email address, first name, and purchase history.
Which field should be designated as the Primary Key for this customer Data Extension?
A
Email Address
B
Customer ID
C
First Name
D
Purchase History

âœ… CORRECT ANSWER: B

Why this is correct: Customer ID serves as the optimal Primary Key because it represents a unique, immutable identifier that maintains data integrity and referential consistency across systems. Unlike email addresses, Customer IDs are designed to remain constant throughout the customer lifecycle, ensuring reliable record identification even when other attributes change. Primary Keys in Data Extensions must enforce uniqueness constraints and provide stable reference points for data relationships, making Customer ID the logical choice for this core identifier role. This approach aligns with database normalization best practices and supports accurate customer journey tracking, personalization, and cross-system data integration.

Why other options are incorrect:

  • Option A: Email addresses can change over time as customers update their contact information, create new email accounts, or experience deliverability issues. Using email address as a Primary Key would create data integrity problems when addresses change, potentially orphaning historical data and breaking relationships between Data Extensions.
  • Option C: First Name is not unique across customers and can change due to legal name changes, cultural preferences, or data entry corrections. Using non-unique fields as Primary Keys violates fundamental database design principles and would cause data import failures and record conflicts.
  • Option D: Purchase History is a complex, multi-valued field that changes frequently as customers make new purchases. It's neither unique nor stable, making it completely unsuitable as a Primary Key. Additionally, transactional data like purchase history should be stored in separate, related tables.

Key Learning Points:

  • Primary Keys must be unique, immutable, and non-null for data integrity
  • Customer-generated identifiers provide better stability than user-provided data
  • Proper Primary Key selection supports data relationships and system integrations
  • SFMC Data Extensions enforce Primary Key constraints to prevent duplicate records

  • Option B - Subscriber Key field: The Subscriber Key field provides a unique identifier that links records in the Data Extension to subscriber records in SFMC's All Subscribers list. This connection enables subscription management, tracking, and compliance features. The Subscriber Key is essential for maintaining subscriber relationships across campaigns and channels.
  • Option D - Primary Key field: Every Data Extension requires a Primary Key field to ensure unique record identification and prevent duplicate entries. The Primary Key maintains data integrity, supports efficient data operations, and enables reliable record updates and deletions during data import processes.

  • Option C: First Name is an optional personalization field that enhances email content but is not required for basic sendable functionality. While valuable for personalization, it's not a technical requirement for creating a functional sendable Data Extension.
  • Option E: Phone Number is an optional contact field that may be useful for omnichannel campaigns but is not required for email sending functionality. It's not a mandatory field for sendable Data Extensions focused on email campaigns.

Key Learning Points:

  • Sendable Data Extensions require specific field designations for email delivery functionality
  • Email Address and Subscriber Key fields create the foundation for SFMC's sending engine
  • Primary Keys are mandatory for all Data Extensions to ensure data integrity
  • SFMC's field designation system connects data structure to platform functionality

Scenario: A marketing team needs to import customer data from their CRM system daily. The data file contains 500,000 records and must be processed automatically without manual intervention.
Which import method is most suitable for this automated, high-volume data import requirement?
A
Manual Import via Import Wizard
B
Automation Studio Import Activity
C
API import calls
D
Email Studio import

âœ… CORRECT ANSWER: B

Why this is correct: Automation Studio Import Activity is specifically engineered for enterprise-scale, automated data processing workflows that require no manual intervention. This tool can handle files with hundreds of thousands of records efficiently, supports various file formats (CSV, TXT, TAB), and can be scheduled to run at specific times or frequencies. The Import Activity integrates seamlessly with file transfer protocols (FTP, SFTP, Enhanced FTP) to automatically retrieve files from external systems, process them according to predefined mapping rules, and update Data Extensions. It also provides comprehensive error handling, logging, and notification capabilities to ensure reliable automated operations. For 500,000 daily records from a CRM system, this represents the most robust and scalable solution.

Why other options are incorrect:

  • Option A: Manual Import via Import Wizard requires human intervention for each import operation, making it unsuitable for daily automated processes. The wizard interface is designed for occasional, smaller imports and would create operational bottlenecks and potential points of failure in a high-volume daily workflow.
  • Option C: API import calls are better suited for real-time, smaller batch imports or when immediate processing is required. While APIs can handle automation, they're not optimized for processing 500,000 records daily and may face rate limiting, timeout issues, and higher resource consumption compared to batch import processes.
  • Option D: Email Studio import functionality is designed for simple subscriber list imports and lacks the enterprise-grade automation, scheduling, and high-volume processing capabilities needed for this CRM integration scenario. It's not built for complex, automated data workflows.

Key Learning Points:

  • Automation Studio Import Activities are optimized for scheduled, high-volume data processing
  • Automated workflows eliminate manual intervention and reduce operational risks
  • Different import methods serve different volume and timing requirements
  • SFMC's Automation Studio provides enterprise-grade data integration capabilities
